# UX 审视规范：基于《Don't Make Me Think》

> 本文档用于指导 AI 模拟真实用户视角，对产品设计进行系统性审视。
> 理论来源：Steve Krug《Don't Make Me Think, Revisited》

---

## 审视原则

在开始审视之前，建立以下用户视角：

- **你是一个普通用户**，不是产品团队成员，没有任何背景知识
- **你很忙**，不会认真阅读任何说明，只会快速扫描页面
- **你不会深思熟虑**，遇到第一个"说得通"的选项就会点击
- **你不会读文档**，靠猜和试错完成操作
- **你随时可以离开**，任何让你费劲的地方都会消耗你的耐心

---

## 一、认知负担审视

**核心标准：页面上的每一个元素，都不应该让用户停下来思考。**

### 1.1 页面是否不言自明？

对每个页面/功能入口提问：

- [ ] 一个从没见过这个产品的人，能否在 **5 秒内** 说出这个页面是做什么的？
- [ ] 页面上有没有让用户产生疑问的文字、图标、或按钮？
- [ ] 用户能否在不读任何说明的情况下完成核心操作？

**判断方法**：列出页面上所有可能产生"这是什么意思？"的元素。每一个问号都是失败。

### 1.2 命名是否清晰？

- [ ] 所有按钮、链接、标签的命名，是否直接说明目标，而不是聪明的创意文案？
- [ ] 功能名称是否使用用户熟悉的词汇，而不是公司内部术语？
- [ ] 有没有让用户需要思考"这个词是什么意思"的命名？

**示例**：
- ❌ "探索更多可能" → ✅ "查看所有功能"
- ❌ "Job-o-Rama" → ✅ "招聘信息"
- ❌ "立即开启您的旅程" → ✅ "注册"

---

## 二、用户行为模拟

**核心标准：设计必须符合用户真实行为，而不是假设中的理想行为。**

### 2.1 扫描测试

用户不读页面，只扫描。模拟扫描行为：

- [ ] 快速浏览页面 3 秒，能否抓住核心信息和主要操作入口？
- [ ] 关键信息（价格、核心功能、行动按钮）是否在扫描中就能被发现？
- [ ] 有没有重要内容藏在大段文字中间，扫描时会被忽略？

### 2.2 "满足即止"测试

用户选第一个"说得通"的选项，不做最优选择：

- [ ] 如果用户点了一个"看起来对但实际上不是最优"的入口，能否轻松发现错误并返回？
- [ ] 导航层级是否清晰，让用户每次点击都有"我在正确路径上"的信心？
- [ ] 链接文字是否给出足够的"信息气味"，让用户知道点击后会到哪里？

### 2.3 "摸索着过"测试

用户靠猜和试错，不读说明：

- [ ] 核心操作流程（注册、购买、完成任务），能否在零引导下完成？
- [ ] 如果用户做错了，错误提示是否清晰告知出了什么问题、怎么修正？
- [ ] 有没有操作是"不试错就找不到入口"的？

---

## 三、视觉设计审视

**核心标准：视觉设计的目的是替用户做预处理，而不是让用户自己去解析。**

### 3.1 视觉层级

- [ ] 页面上最重要的内容，是否在视觉上最突出（更大、更粗、更醒目）？
- [ ] 相关的内容是否在视觉上相邻、归组？
- [ ] 层级关系是否通过嵌套、缩进、大小差异清晰体现？
- [ ] 如果把页面"眯眼看"，重要内容是否依然突出？

### 3.2 惯例遵守

- [ ] 产品是否遵循用户已熟悉的设计惯例（Logo 在左上、导航在顶部/左侧、搜索框在右上）？
- [ ] 有没有打破惯例的设计？如果有，它是否比惯例**明显更易理解**？
- [ ] 所有重新发明的交互模式，是否都经过了用户验证？

**原则**：创新的举证责任在设计师身上，不在用户身上。

### 3.3 可点击元素识别

- [ ] 所有可点击的按钮和链接，静态状态下是否一眼可辨？（不依赖 hover）
- [ ] 有没有用同一颜色表示链接和不可点击的标题？
- [ ] 触摸端：没有 hover 状态，可点击元素是否依然清晰？

### 3.4 噪音控制

- [ ] 页面上有没有"所有东西都在抢注意力"的情况？
- [ ] 有没有没有对齐网格、看起来像被翻乱的区域？
- [ ] 信息密度是否过高？去掉一半内容后，页面是否依然传达完整信息？

**默认假设**：页面上所有元素都是噪音，除非它能证明自己有存在价值。

---

## 四、文案审视

**核心标准：删掉一半词，再删掉剩下的一半。**

### 4.1 文字量

- [ ] 每个页面的文字，删掉 50% 后，信息是否依然完整？
- [ ] 有没有段落可以拆成更短的段落或项目列表？
- [ ] 有没有用三句话能说清楚的事，写了十句话？

### 4.2 Happy Talk 检测

对以下类型的文字提出质疑：

- [ ] "欢迎来到……，我们很高兴为您提供……"式的开场白 → **全部删掉**
- [ ] "我们致力于为用户创造……"式的使命宣言 → **删掉**
- [ ] 任何读的时候脑子里会产生"blah blah blah"感觉的文字 → **删掉**

### 4.3 说明文字检测

- [ ] 有没有大段操作说明？（用户不会在遇到问题之前读说明）
- [ ] 表单、流程页面上，能否去掉所有"请在此处填写……"式的提示？
- [ ] 如果说明无法删除，是否已压缩到极限？

---

## 五、导航审视

**核心标准：导航不只是功能，它就是产品本身。**

### 5.1 行李箱测试

随机打开产品任意一个深层页面，不仔细看，能否立即回答：

- [ ] 这是哪个产品/网站？（Logo/品牌标识）
- [ ] 我在哪个页面？（页面标题）
- [ ] 产品的主要分区有哪些？（主导航）
- [ ] 当前层级的选项有哪些？（本地导航）
- [ ] 我在整体结构中的位置？（"你在这里"指示器/面包屑）
- [ ] 如何搜索？（搜索入口）

**判断标准**：以上六个问题，必须不需要仔细研究就能秒答。

### 5.2 持久导航检查

- [ ] Logo/品牌标识是否出现在每个页面，且点击可返回首页？
- [ ] 主导航是否在每个页面保持一致的位置和样式？
- [ ] 实用工具（登录、购物车、帮助）是否始终可见且不超过 4-5 个？
- [ ] 搜索框是否标注为"Search"或使用放大镜图标，且不加额外说明？

### 5.3 页面命名

- [ ] 每个页面是否有清晰的页面标题？
- [ ] 页面标题是否与用户点击的链接文字匹配？（违反此规则会损害信任）
- [ ] 页面标题是否在视觉上最突出？

### 5.4 深层导航

- [ ] 第三层以下的页面，导航是否和顶层一样完整？
- [ ] 用户在深层页面是否始终能看到自己在哪里、能去哪里？

---

## 六、首页审视

**核心标准：用户进入首页的前几秒，决定他们对整个产品的理解框架。**

### 6.1 大局传达

用户进入首页后，必须能在几秒内回答：

- [ ] 这是什么产品？
- [ ] 它能为我做什么？
- [ ] 这里有什么内容/功能？
- [ ] 我从哪里开始？

### 6.2 标语检查

- [ ] 产品是否有清晰的标语（Tagline），位于 Logo 附近？
- [ ] 标语是否传递了价值主张，而不是鼓励性口号？
  - ❌ 口号："让美好发生" / "连接世界"
  - ✅ 标语："全网最低价机票比价" / "专为设计师的项目管理工具"
- [ ] 标语是否足够具体，以至于只有你们能用，竞品不能用？

### 6.3 首页内容优先级

- [ ] 首页最重要的内容（用户最常做的事）是否最显眼？
- [ ] 首页有没有被各方利益方塞满，导致用户不知道从哪里开始？
- [ ] 有没有"捍卫首页简洁度"的机制或负责人？

---

## 七、交互与表单审视

**核心标准：用户不应该被惩罚，因为他们不按你期望的方式填写。**

### 7.1 选择简化

- [ ] 所有需要用户做选择的地方，选项是否"无脑"——不需要思考就能选对？
- [ ] 有没有需要用户搞清楚区别才能选择的选项？如果有，必须提供清晰说明
- [ ] 复杂的选择是否拆分成多步骤，每步只问一件事？

### 7.2 表单

- [ ] 表单是否只收集完成任务必须的信息，没有多余字段？
- [ ] 有没有强制用户按特定格式输入的字段？（电话号码、日期、信用卡号）
  - ❌ "请输入不含空格的信用卡号"
  - ✅ 自动处理格式，用户怎么输都行
- [ ] 错误提示是否清晰说明"哪里错了"和"怎么改"？

### 7.3 点击路径

- [ ] 核心操作路径的每一步点击，是否都不需要思考？
- [ ] 有没有步骤让用户产生"我不确定这一步对不对"的感觉？
- [ ] 如果用户走错了，返回或纠错的代价是否足够小？

---

## 八、善意水库审视

**核心标准：产品的每个设计决策，都在消耗或补充用户的善意。**

### 8.1 会消耗善意的设计（逐项检查）

- [ ] **信息藏匿**：定价、运费、重要限制条件，是否被藏到流程深处？
- [ ] **格式惩罚**：有没有因为用户格式不对而拒绝提交的表单？
- [ ] **强制注册**：是否在用户还没得到价值之前就要求注册？
- [ ] **虚假关怀**：有没有系统性的假关怀话术（"您的反馈对我们非常重要"）？
- [ ] **无用内容**：首页或关键页面是否有大量营销图片占用用户时间？
- [ ] **隐藏客服**：客服联系方式是否需要多次点击才能找到？

### 8.2 会补充善意的设计（逐项检查）

- [ ] 用户最常做的事，是否在产品中最容易完成？
- [ ] 重要信息（价格、限制、已知问题）是否主动告知，而不是等用户踩坑？
- [ ] 能否帮用户减少步骤？（例：确认邮件里直接附上可点击的追踪链接）
- [ ] 出错时，恢复路径是否简单清晰？
- [ ] 做不到用户期望的事，是否有明确说明和替代方案？

---

## 九、移动端专项审视

**在以上所有审视的基础上，额外检查：**

- [ ] 触摸端没有 hover：所有可点击元素在静态下是否清晰可辨？
- [ ] 核心功能是否在最少的点击内可达？
- [ ] 响应式设计：在手机屏幕上显示时，关键信息是否依然不被截断？
- [ ] 页面加载速度：在弱网环境下，是否依然可用？
- [ ] 首次使用引导：新用户做完引导后，能否记住核心操作？

---

## 十、测试建议

以上审视发现的问题，优先级排序建议：

**P0（必须修）**：导致用户完全无法完成核心任务的问题
**P1（应该修）**：让用户困惑超过 3 秒、或需要猜测的问题
**P2（建议修）**：消耗善意但不阻碍任务完成的问题
**P3（可以修）**：视觉噪音、冗余文案等优化项

**验证方法**：找 3 个从未见过该产品的用户，让他们完成核心任务，观察卡点。不要帮助，不要引导，只观察。
